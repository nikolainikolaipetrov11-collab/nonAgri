# AGENTS.md

本文件用于约束 AI 助手在本项目中的工作方式。所有自动化修改、代码检查、脚本运行和说明输出，都应遵守这里的规则。

## 基本原则

- 优先保护用户已有文件和数据，不要擅自覆盖、还原或删除用户已经做过的修改。
- 修改代码前，先阅读相关文件，理解现有写法和项目结构，再进行小范围、可解释的修改。
- 用户的专业知识较弱，因此每次给出改动、优化或排错建议时，都要用通俗语言解释原因、影响和使用方法。
- 代码修改应尽量保持简单直接，避免引入不必要的新框架、新目录或复杂抽象。
- 如果任务存在风险、不确定性较高，或者需要用户确认业务含义，应先说明情况，再继续操作。

## 文件删除安全规则

禁止批量删除文件或目录。

不要使用以下命令：

```powershell
del /s
rd /s
rmdir /s
Remove-Item -Recurse
rm -rf
```

需要删除文件时，只能一次删除一个明确路径的文件，例如：

```powershell
Remove-Item "C:\path\to\file.txt"
```

如果确实需要批量删除文件，应停止操作，并让用户手动删除，或先向用户明确说明需要删除哪些文件、为什么删除，再等待用户确认。

## 代码修改要求

- 修改前先查看相关文件，不要凭空猜测代码结构。
- 只修改和当前任务直接相关的文件，避免顺手重构无关代码。
- 如果发现已有未提交或不属于本次任务的改动，不要回退、覆盖或删除它们。
- 新增逻辑要尽量配合项目原有风格，例如函数命名、配置读取方式、日志输出方式等。
- 对复杂逻辑添加必要注释，帮助用户理解为什么这样写；不要添加无意义注释。
- 修改完成后，应说明改了哪些文件、解决了什么问题，以及用户下一步如何运行或验证。

## 命令和脚本运行要求

- 优先使用只读命令查看项目，例如 `Get-ChildItem`、`Get-Content`、`rg`。
- 查找文本或文件时优先使用 `rg`，如果不可用再使用 PowerShell 自带命令。
- 运行脚本前，尽量确认它不会批量删除、覆盖或移动文件。
- 如果命令可能联网、访问外部服务、写入系统目录或执行高风险操作，应先向用户说明。
- 不要运行破坏性 Git 命令，例如 `git reset --hard` 或强制覆盖工作区，除非用户明确要求。

## Python 项目约定

- 优先使用项目已有的配置文件和依赖，例如 `config.toml`、`requirements.txt`、现有 Python 脚本中的工具函数。
- 修改 Python 代码时，保持变量名清晰，避免过度简写。
- 对输入路径、输出路径、数据列名、空间参考、文件格式等容易出错的位置，尽量加入清晰的检查或报错信息。
- 如果新增依赖，应说明为什么需要它，并更新依赖文件。
- 如果能用小范围测试、示例命令或 dry-run 方式验证，就优先给出验证方法。

## 给用户的说明方式

- 使用中文回复。
- 解释时尽量避免术语堆叠；如果必须使用专业词，要顺带说明它是什么意思。
- 对代码改动要说明“改了什么”“为什么改”“怎么验证”。
- 如果用户需要手动操作，步骤要清楚、具体，最好给出可以直接复制的命令。
- 如果没有完成某一步，例如测试没跑成功，要明确告诉用户原因和风险。
